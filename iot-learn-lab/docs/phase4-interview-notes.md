## 面试题 1：滚动 / 蓝绿 / 金丝雀优缺点？

### 我的初答（摘要）

- **滚动：** 逐渐替换；出问题回滚较慢；操作简单（优点不太清楚）。
- **蓝绿：** 一次性替换；回滚快；但新版本未经验证就接全量流量，有风险。
- **金丝雀：** 新旧并存，少量流量验证；稳定但流程麻烦，需要观察阶段。

### 精炼结论（面试版）

| 策略 | 流量怎么切 | 资源 | 回滚速度 | 验证程度 | 本 lab |
|------|------------|------|----------|----------|--------|
| **滚动** | 逐实例替换 | 低 | 中（逐批回退） | 中 | 未完整模拟 |
| **蓝绿** | 一次性全切 | 高（双倍） | 快（切回旧环境） | 切换前有限 | 可用 0/100 权重近似 |
| **金丝雀** | 按权重渐进 | 中 | 快（降权/切回） | 高（真实小流量） | **C2–C4 核心** |

### 滚动发布（Rolling Update）

**机制：** 逐个或逐批替换实例（如 10 台先换 2 台，再换下一批）。

| 优点 | 缺点 |
|------|------|
| **资源占用低**，不需双倍机器 | 发布期**新旧版本共存**，行为可能不一致 |
| K8s / CI/CD **原生支持**，操作简单 | 回滚需逐批换回，**比蓝绿慢** |
| 适合**小改动、低风险**常规发布 | **难做精细灰度**（如只让 5% 用户试新版本） |

**面试一句：** 滚动适合日常低风险发布；不适合需要「先小流量验证再全量」的高风险变更。

### 蓝绿发布（Blue-Green）

**机制：** 绿环境（新版本）全量部署好 → **一切流量从蓝切到绿**（DNS / LB / 网关）。

| 优点 | 缺点 |
|------|------|
| **回滚极快**：切回蓝环境即可 | **资源翻倍**（蓝绿各一套全量） |
| 切换瞬间完成，运维路径清晰 | **切换前未接真实全量流量**，预发再充分也可能漏问题 |
| 适合大版本、配合数据库迁移 | DB/schema 不兼容时蓝绿**并不简单** |
| 用户感知为「一次性升级」 | 切换瞬间若绿有问题，**100% 流量同时踩坑** |

**面试一句：** 蓝绿换的是**切换速度与回滚速度**，不是**验证充分性**。高风险变更常 **蓝绿 + 金丝雀** 组合。

### 金丝雀发布（Canary）

**机制：** 新旧并存，按权重把少量流量（如 5%～10%）打到新版本，观测正常后再 **10% → 30% → 50% → 100%** 逐步放大。

| 优点 | 缺点 |
|------|------|
| **用真实流量验证**，爆炸半径可控 | **流程复杂**：权重、监控、回滚脚本、观察窗口 |
| 指标异常可**快速降权或回滚** | **观察期拉长**（常见 15～30min+） |
| 适合核心链路、高流量服务 | 需 **按版本打点**（`version=v1/v2`） |
| 与 APISIX、Nacos 配合自然 | 小流量下错误率统计波动大 |

**面试一句：** 金丝雀用**复杂度**换**发布安全**；Phase 4 的 C2～C4 就是在练这个。

### 自评

| 项 | 得分感 | 待加强 |
|----|--------|--------|
| 三策略对比 | 7/10 | 滚动补「省资源、平台原生」；金丝雀强调「真实流量 + 可控影响面」 |

## 面试题 2：发布失败怎么发现？

### 我的初答（摘要）

- 通过**错误率**观察；新版本上线后错误率上升则很可能有问题，须回退。
- 查看新服务**日志**，建立**告警**；错误率上升且持续符合阈值则触发回滚。

### 精炼结论（面试版）

发布失败 = **可观测性发现 + 明确 SLO/阈值 + 回滚预案**；不只看错误率一个数。

### 看什么（按优先级）

| 信号 | 看什么 | 为什么 |
|------|--------|--------|
| **错误率** | 5xx 比例、按 `version` 拆分 | 金丝雀时**总错误率**可能只略升，要看 **v2 >> v1** |
| **延迟** | P95/P99、超时率 | 内存泄漏、慢 SQL 常先表现为**尾延迟**，错误率仍为 0 |
| **流量** | QPS 是否按权重分布 | 分流配错会导致「以为金丝雀，实际全打 v2」 |
| **业务指标** | 上报成功率等 | 技术指标正常，业务仍可能异常 |
| **日志 / Trace** | 新版本 stack trace、新异常 | 定位根因，区分发布问题 vs 偶发 |
| **告警** | 多条件、持续时长 | 避免单次抖动误回滚 |

### 本 lab PromQL

```promql
# v2 错误率（金丝雀关键）
sum(rate(http_server_requests_seconds_count{application="device-report-service",version="v2",status=~"5.."}[5m]))
/
sum(rate(http_server_requests_seconds_count{application="device-report-service",version="v2"}[5m]))

# 总错误率（全局是否可控）
sum(rate(http_server_requests_seconds_count{application="device-report-service",status=~"5.."}[5m]))
/
sum(rate(http_server_requests_seconds_count{application="device-report-service"}[5m]))

# 分版本 QPS（验证 9:1 分流）
sum(rate(http_server_requests_seconds_count{version="v1"}[1m]))
sum(rate(http_server_requests_seconds_count{version="v2"}[1m]))
```

**金丝雀 10% 时：** 总错误率 ≈ `0.1 × v2 错误率`（若 v1 稳定）。

### 告警 / 回滚策略（加分项）

1. **按版本看**（例：v2 5xx > 5% **持续 5min**）
2. **对比基线**（相对发布前 v1 错误率突增）
3. **多信号确认**（错误率 + P99 + 日志新异常）
4. **自动 vs 人工**：lab 里 C4 执行 `bootstrap-canary-rollback.sh`；生产可能是告警 → 一键回滚或自动降权

### 面试话术

> 金丝雀发布不能只看全局错误率。10% 流量到新版本时，全局可能只略升，必须按 `version` 拆开看。v2 错误率异常而 v1 正常，说明是新版本问题，应降权或回滚，而不是等全量中招。

### 自评

| 项 | 得分感 | 待加强 |
|----|--------|--------|
| 发布失败发现 | 8/10 | 加 P99、按 version 拆分、持续时长告警 |

## 面试题 3：金丝雀与 Sentinel 熔断关系？

### 我的初答（摘要）

- **金丝雀：** 版本发布，少部分流量到新版本；有错可快速回滚或告警自动回滚。
- **Sentinel：** 下游宕机/错误时熔断，走兜底，不让请求继续打下游。

### 精炼结论（面试版）

| 维度 | 金丝雀（发布策略） | Sentinel 熔断（运行时韧性） |
|------|-------------------|---------------------------|
| **目的** | 控制**新版本**上线风险 | 保护系统免受**依赖故障**拖累 |
| **触发** | **人/发布流程**主动调权重 | **自动**（错误率、慢调用、RT） |
| **作用对象** | 新旧**版本实例**分流 | 调用**下游服务**（如 command-dispatch） |
| **典型动作** | 90/10 → 回滚 100% v1 | 熔断打开 → 快速失败 / Feign fallback |
| **时间尺度** | 发布窗口（分钟～小时） | 运行时（秒～分钟级自动恢复） |
| **本 lab Phase** | Phase 4（C2–C4） | Phase 3（R4/R5/R6） |

### 关系（不是二选一，可并存）

- **金丝雀**回答：「要不要让更多用户用**新版本**？」
- **Sentinel**回答：「下游挂了，**当前版本**怎么不被拖死？」

**可同时存在：**

- 金丝雀 10% 流量到 v2；
- v2 调 command-dispatch 时仍走 Sentinel；下游熔断 → fallback，**不等于**金丝雀失败，但要**按 version** 分析是 v1 还是 v2 在触发降级。

**Phase 4 实验建议：** APISIX 限流插件设为 `none`，避免与发布观测混淆；Sentinel 仍可开启，分析时区分「版本 bug」vs「下游故障」。

### 面试话术

> 金丝雀解决**变更风险**——新版本好不好，用少量真实流量验证，坏了就降权回滚。Sentinel 解决**运行风险**——依赖已经坏了，限制调用、快速失败、降级兜底，防止雪崩。一个管发布时怎么切流量，一个管运行时怎么扛故障。

### 自评

| 项 | 得分感 | 待加强 |
|----|--------|--------|
| 金丝雀 vs Sentinel | 8/10 | 已抓住场景差异；可强调「可并存」及 Phase 分工 |

## Nacos 热更新踩坑（C6 附录）

| 症状 | 根因 |
|------|------|
| 有 `Refresh keys changed` 但行为不变 | `@ConditionalOnProperty` 或构造器 `final` 缓存旧值 → 需 `@RefreshScope` |
| 完全拉不到配置 | Data ID 须为 `device-report-v2-app.yaml`（与 import 一致） |
| `optional:` 无报错 | 拉取失败被静默忽略，排查时临时去掉 `optional:` |
